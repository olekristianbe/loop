# Loop v1.4.0

Loop 1.4.0 makes Plan, Build, and Review feel like first-class parts of the same
conversation while letting each provider do what it is best at. Loop supplies
the protocol, repository context, durable artifacts, validation, and desktop
controls; Claude Code, Codex, or Cursor performs the work and can use its native
agents or parallel workflows internally.

These notes cover the complete change from v1.3.0. They include the workflow
and product improvements developed for v1.3.1, which was never published as a
stable public release, plus the later provider-native architecture cutover and
reliability work.

## Provider-native runs

- Plan now progresses through an ordinary visible provider conversation, while
  Build and Deep Review each dispatch one visible coordinator turn. None uses
  a tree of Loop-created hidden task threads.
- The provider decides when native subagents or parallel work are useful. The
  main thread remains the coordinator and final synthesizer, so questions,
  decisions, progress, and results stay in one place.
- Claude, Codex, and Cursor activity is normalized into shared agent and tool
  events. Native agents appear as compact expandable timeline cards, while
  unknown provider events are retained for diagnostics without flooding the
  work log.
- For Build and Deep Review, a provider-native run projector keeps the common
  run card and durable `.loop/runs/<run-id>/run.json` history without
  reintroducing a Loop-owned task scheduler.
- The intermediate custom Workflow Engine and its engine-only task skills were
  removed before this release. The remaining run state is a progress and
  history surface, not a dependency-graph executor.
- Removed Loop's Skills settings, managed-skill registry, and automatic
  host-folder projection as part of the provider-native cutover. Loop no longer
  installs or manages provider skills; formal Plan, Build, and Review behavior
  comes from each mode's composed run protocol.

## Plan

- Plan Mode now follows a fixed conversation protocol: intake, repository and
  documentation research, synthesis, focused interview, explicit in-chat
  confirmation, write, and self-check.
- New plans are canonical Markdown at
  `.loop/plans/<plan-id>/plan.md`. Each plan has a goal, stable `W<number>` work
  items with acceptance and verification text, and a definition of done.
- Loop validates the latest plan live. Build is disabled when the plan is
  missing or structurally invalid, and the sidebar shows the concrete issues.
- A newly written invalid plan receives exactly one visible repair request.
  Loop also detects when a provider claims to have written a plan but no valid
  file change was observed.
- Plan Mode writes only under `.loop/plans/` and only after explicit user
  confirmation. It no longer writes `.planner/session.json` or runs hidden
  forward-stage children.

## Build

- Build runs the full plan, selected work items, or a custom slice as one
  visible coordinator turn. The provider can distribute independent
  implementation, research, and verification through its native mechanisms.
- Loop prepares the selected branch before dispatch and now honors every Build
  option: auto-commit, push after commit, prevent sleep, advisory network
  access, and Stop.
- Auto-commit uses a post-turn safety net for leftover changes; push only runs
  when enabled and the branch is ahead. It requires a clean source baseline,
  stages only project source (never `.loop` or `.planner` artifacts), restores
  the index without discarding worktree changes on failure, and treats commit
  or push errors as Build failures. Provider subagents are instructed not to
  commit or push independently.
- Stop interrupts the active provider turn and produces a cancelled summary.
  Starts are guarded so write-capable runs cannot collide in the same effective
  worktree, and late completion cannot commit, push, repair, or overwrite a
  stopped run.
- Every run writes
  `.loop/runs/<run-id>/reports/build-report.md` with changed files, non-visual
  verification, visual verification, evidence, open issues, and result. Loop
  validates the report, permits one repair request, and derives
  `.loop/runs/<run-id>/summary.json` from the result.
- Build drafts are promoted to real server threads before dispatch, and
  completion state is scoped to the thread that actually ran the build. This
  removes phantom completion panels in fresh drafts.

## Review

- Review starts from an empty-by-default builder with eight check categories
  and targeted scopes, including uncommitted changes, branch differences, and
  the entire codebase.
- Deep Review sends the selected scope and checks through one read-only
  provider-native protocol. Substantive reviews can fan out to native
  specialists, then re-check and consolidate their findings into one result.
- Read-only mode is enforced at each provider boundary for both Normal and Deep
  Review, independent of the thread's ordinary permission setting.
- The result separates High, Medium, and Low findings, verified clean areas,
  residual risk, and the verdict. Deep Review persists it at
  `.loop/runs/<run-id>/reports/review.md`; Loop validates the exact section
  contract and only marks the run complete after the report is durable.
- Review results can be fixed in Chat or handed to Plan Mode through a new
  editable draft prefilled with the consolidated review.

## Workflow and workspace improvements

- Added draft-scoped context notes that remain visible in the composer instead
  of disappearing into hidden prompt context.
- Unified automated verification and human checks in the Activity / Artifacts
  surface, including editable status and notes plus a feedback-to-chat handoff.
- Added optional global model and effort overrides for Chat, Plan, Review, and
  Build. The effective override now controls the complete dispatch context,
  including provider-specific image validation, effort, and the final model
  selection. Unset modes inherit the visible thread selection.
- Added opt-in desktop notifications for completed, failed, or input-blocked
  runs while Loop is in the background.
- Added per-project browser viewport presets and richer preview-note metadata.
- Added **Change Path** for projects, with merge/history preservation when the
  destination already exists. Backend validation now requires the selected
  merge target to own that normalized path and rejects duplicate ownership
  atomically.

## Providers and models

- Claude Code can now receive supported image attachments through its native
  stream protocol; unsupported image types are rejected before dispatch.
- Claude subagent lifecycle events and Codex collaboration events now drive
  real native-agent cards. Claude sidechain output is kept out of the main
  assistant message, and repeated/cumulative events are deduplicated.
- Cursor tool calls now use the same start/update/complete lifecycle as the
  other providers, with an honest single-session fallback when task-level
  activity is unavailable.
- Cursor model and option-selection failures are surfaced instead of silently
  falling back to another model; Grok 4.5 only advertises controls Loop can
  apply deterministically.
- Codex command-output deltas are folded into their existing command row instead
  of generating a stream of anonymous work-log entries.
- The built-in model catalog adds Claude Fable 5, Cursor Grok 4.5, and the Codex
  GPT-5.6 Sol, Terra, and Luna variants. Custom model slugs remain supported.

## Reliability and polish

- Fixed the Servers tab so detected HTTP listeners open directly in Loop's
  browser, terminal-derived servers stay scoped to the active project or
  worktree, and stop actions revalidate project ownership before signaling.
- Fixed commit-message generation for untracked-only source changes. The Git
  dialog now passes its exact refreshed selection, excludes Loop-owned `.loop`
  artifacts, preserves unrelated staged work, and publishes the selected tree
  with branch/index compare-and-swap guards instead of sweeping the shared
  index.
- Finished the native thread-terminal drawer plumbing so open/focus actions and
  session state stay stable in the Tauri desktop runtime.
- Fixed terminal sessions remounting, blinking, and losing focus on every chat
  streaming update.
- Fixed a React 19 maximum-update-depth loop when a thread had non-empty
  proposed plans by keeping external-store snapshots referentially stable.
- Fixed the same React 19 snapshot contract for object-valued local preferences,
  which could loop after a project script stored its last-invoked state.
- Kept project-terminal launch context through bootstrap cleanup so the shared
  project terminal opens in the intended directory instead of a thread worktree.
- Fixed build stop/restart races, stale summaries after daemon restart,
  cross-thread build-state replay, and second-start collisions.
- Gave every Build and Review a unique durable run ID, preserved repeated-run
  history on one thread, and made thread subscriptions recover the latest
  terminal run from `.loop/runs` after a daemon restart.
- Rejected symlinked or escaping `.loop` artifact trees before formal runs and
  routed plan validation, migration, Build, Review, summary, and run-state I/O
  through containment-checked atomic helpers.
- Made provider timeouts interrupt and force-stop the child before finalizing,
  and retained a visible, retryable Failed run when terminal-state persistence
  cannot complete.
- Fixed empty run cards showing `0/0 complete` before the provider had spawned
  native work.
- Fixed sidebar crashes when older activity snapshots omitted task arrays and
  kept legacy hidden workflow children out of normal navigation.
- Preserved preview bounds during desktop initialization and tightened preview
  capture errors, selection-note positioning, diff rendering, and work-log
  filtering.
- Removed React Compiler bailouts that affected component stability and
  terminal behavior without adding rule suppressions or code workarounds.
- Forbid unsafe Rust workspace-wide and restored warning-free clippy, workspace
  test, typecheck, and lint baselines during the development cycle.
- Resolved the production and development dependency audit while keeping
  security overrides within the dependency majors already supported by Loop.
- Extended the legacy macOS updater manifest with DMG metadata while preserving
  the native Tauri update path. The final distributed DMG is now separately
  notarized, stapled, Gatekeeper-assessed, and verified before its blockmap and
  manifest hashes are generated.

## Artifacts and migration

New work uses these project-local paths:

```text
.loop/plans/<plan-id>/plan.md
.loop/plans/<plan-id>/validation.json
.loop/runs/<projected-run-id>/run.json
.loop/runs/<build-id>/summary.json
.loop/runs/<build-id>/reports/build-report.md
.loop/runs/<build-id>/screenshots/
.loop/runs/<review-id>/reports/review.md
```

`.planner` is legacy input only. On first use, Loop can copy existing planner
state and history into `.loop` non-destructively; it never deletes or rewrites
the original `.planner` files. New Plan and Build flows do not write there.

## Shipping notes

- Loop v1.4.0 is macOS 15+ and Apple silicon only.
- At least one authenticated local provider runtime is required: Claude Code,
  Codex, or Cursor Agent with ACP support.
- The stable public update feed remains compatible with both older Electron
  installations and current Tauri installations.
